# ARC42 - Архитектурная документация

## 1. Введение

> Введение содержит краткое описание всего документа и включает в себя цель, область применения, определения, сокращения, аббревиатуры, ссылки и обзор данного документа.

### 1.1. Что такое ARC42?

**ARC42** — это бесплатный шаблон и методология для документирования архитектуры программного обеспечения.
Название происходит от "Architecture" и "42" (отсылка к "Ответу на главный вопрос жизни,
вселенной и всего такого" из романа Дугласа Адамса).

#### 1.1.1. Зачем нужна архитектурная документация?

- **Коммуникация**: Помогает команде и заинтересованным сторонам понимать систему
- **Принятие решений**: Документирует важные архитектурные решения и их обоснование
- **Онбординг**: Ускоряет включение новых участников в проект
- **Поддержка**: Облегчает сопровождение и развитие системы
- **Соответствие стандартам**: Помогает соответствовать корпоративным и отраслевым требованиям

#### 1.1.2. Принципы ARC42

- **Практичность** - фокус на практически полезной информации
- **Структурированность** - четкая организация информации
- **Масштабируемость** - подходит для проектов любого размера
- **Гибкость** - можно адаптировать под нужды проекта
- **Полнота** - покрывает все аспекты архитектуры

## 2. Структура ARC42

ARC42 состоит из 12 основных разделов, каждый из которых фокусируется на определенном аспекте архитектуры. Разделы организованы по принципу "от общего к частному" и "от статики к динамике".

### 2.1. [Introduction and Goals](./01_Introduction_and_Goals.md)
Краткое описание требований, движущих сил и целей проекта.  
Помогает понять "зачем" создается система и какие цели преследует.

- Краткое описание требований и движущих сил
- 3-5 основных целей качества для архитектуры
- Таблицу заинтересованных сторон с их ожиданиями
- Аннотацию требований

Ответственная проектная роль: **Project Owner**

### 2.2. [Constraints](./02_Constraints.md)
Все ограничения, которые влияют на архитектурные решения.  
Помогает понять границы, в которых можно принимать решения.

- Технические ограничения (технологии, платформы)
- Организационные ограничения (бюджет, время, команда)
- Правовые ограничения (соответствие стандартам)
- Операционные ограничения (инфраструктура, безопасность)

Ответственная проектная роль: **Project Owner**

### 2.3. [Context and Scope](./03_Context_and_Scope.md)
Определяет границы системы и её взаимодействие с внешним миром.  
Помогает понять "что" делает система и с кем взаимодействует.

- Диаграмму контекста системы
- Описание внешних интерфейсов
- Границы системы (что входит, что не входит)
- Взаимодействие с пользователями и внешними системами

Ответственная проектная роль: **Architect**

### 2.4. Solution Strategy
Высокоуровневые стратегические решения по архитектуре.  
Определяет "как" в общих чертах будет реализована система.

- Технологический стек
- Архитектурные паттерны и стили
- Подходы к достижению целей качества
- Организационные решения

Ответственная проектная роль: **Architect**

### 2.5. Building Block View
Статическая структура системы - из чего она состоит.  
Показывает структуру системы "в покое".

- Иерархию строительных блоков (компонентов)
- Интерфейсы между компонентами
- Ответственности каждого компонента
- Декомпозицию до нужного уровня детализации

Ответственная проектная роль: **Architect**

### 2.6. Runtime View
Динамическое поведение системы - как компоненты взаимодействуют.  
Показывает как система работает "в движении".

- Сценарии важных вариантов использования
- Последовательности взаимодействий
- Обработка ошибок и исключений
- Административные сценарии

Ответственная проектная роль: **Analyst**

### 2.7. Deployment View
Техническая инфраструктура и развертывание системы.  
Показывает где и как система будет работать.

- Диаграммы инфраструктуры
- Сопоставление компонентов с серверами/контейнерами
- Требования к средам (dev, test, prod)
- Сетевую топологию

Ответственная проектная роль: **DevOps**

### 2.8. Crosscutting Concepts
Общие принципы и подходы, применяемые во всей системе.  
Обеспечивает единообразие подходов в разных частях системы.

- Модели предметной области
- Архитектурные паттерны
- Правила использования технологий
- Стандарты кодирования и проектирования
- Принципы безопасности и логирования

Ответственная проектная роль: **TechLead**

### 2.9. Architectural Decisions
Документирование важных архитектурных решений и их обоснование.  
Сохраняет контекст принятых решений для будущих поколений разработчиков.

- Критические архитектурные решения
- Альтернативы, которые рассматривались
- Обоснование выбора
- Последствия принятых решений

Ответственная проектная роль: **Architect**

### 2.10. Quality Requirements
Требования к качеству системы в виде измеримых сценариев.  
Определяет критерии успеха системы.

- Сценарии производительности
- Требования к безопасности
- Требования к надежности
- Требования к масштабируемости
- Дерево качества для обзора

Ответственная проектная роль: **TestLead**

### 2.11. Risks and Technical Debt
Известные риски и техническая задолженность.  
Помогает планировать работу по улучшению системы.

- Технические риски и их влияние
- Техническая задолженность
- Проблемы, вызывающие недовольство команды
- Планы по устранению рисков

Ответственная проектная роль: **Project Owner**

### 2.12. Glossary
Словарь важных терминов проекта.  
Обеспечивает единое понимание терминологии всеми участниками.

- Определения бизнес-терминов
- Технические термины
- Сокращения и аббревиатуры
- Специфичные для проекта понятия

Ответственная проектная роль: **Analyst**
